# Portfolio 2026 — Règles always-on

Ces règles s'appliquent à toute session Claude sur ce projet. Sans exception.

## Ce que Claude ne doit JAMAIS faire

- Ajouter Tailwind ou un autre framework CSS utility-first.
- Modifier `styles/tokens/primitives.css` ou `styles/tokens/semantic.css` sans discussion préalable.
- Mettre une valeur hex, px, rem, ms, ou font-family en dur dans un composant ou CSS module — toujours via `var(--token-*)`.
- Installer une dépendance (pnpm add) sans demander.
- Skip un hook git (`--no-verify`) ou forcer un push.
- Ajouter une attribution IA / "Generated by Claude" dans un commit.
- Écrire des commentaires WHAT (le code se lit). Seul le WHY non-évident mérite un commentaire.

## Organisation des fichiers

- Composants React : `src/components/{os,window,windows,folder,ui,typography}/<Name>/<Name>.tsx` + `.module.css` colocalisés.
- Store Zustand : `src/stores/windowStore.ts` (store unique pour toutes les fenêtres).
- Contenu projets : MDX statique dans `content/projects/<slug>.mdx` avec frontmatter (`title`, `subtitle`, `tags[]`, `coverImages[]`, `order`).
- Assets : `public/{fonts,projects,icons}/`.
- Path alias : `@/*` → `./src/*` (jamais de chemins relatifs profonds type `../../../`).

## Design tokens (règle principale du projet)

Hiérarchie obligatoire à 3 couches :

```
primitives.css → semantic.css → components.css → themes.css
```

- Un composant consomme **uniquement** `components.css` ou `semantic.css` — jamais `primitives.css` directement.
- Ajouter une nouvelle var dans `components.css` **avant** de l'utiliser dans un `.module.css`.
- Nommer les vars composant par leur rôle : `--window-bar-close`, `--dock-active-indicator-color`, `--button-plain-bg`.

## Style de code

- TypeScript strict : interdire `any`. Utiliser `unknown` + narrowing aux frontières.
- Props React : `interface` nommée. Pas de `React.FC`.
- Immutabilité : spread operator, `Readonly<T>`, jamais de mutation.
- Fichiers : 200-400 lignes idéal, 800 max. Découper dès que ça déborde.
- Commits en Conventional Commits **français** : `feat:`, `fix:`, `docs:`, `chore:`, `test:`, `refactor:`, `perf:`, `style:`.
- Langue des commentaires et docs : français. Identifiants et code : anglais.

## Tests

- Unit (Vitest + Testing Library) : `tests/unit/**/*.{test,spec}.{ts,tsx}`.
- E2E (Playwright, chromium + mobile-safari) : `tests/e2e/**/*.spec.ts`.
- Couverture cible : 80%+ sur les stores, hooks, composants UI purs.
- TDD quand possible : test rouge → implémentation minimale → refactor.
- 3 parcours E2E golden obligatoires (Phase 9) :
  1. Landing → click terminal "curriculum_vitae.html" → CV s'ouvre
  2. Click folder "EDF" → ProjectWindow EDF s'ouvre avec tags
  3. Drag MainWindow → position persiste après reload (sessionStorage)

## Verification avant chaque commit de phase

```bash
pnpm typecheck && pnpm lint && pnpm test:run && pnpm build
```

Les 4 doivent être verts. Si Phase ≥ 4, ajouter `pnpm test:e2e` une fois le golden path en place.
